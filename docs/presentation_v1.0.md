# Synapse-V 임원 발표 스크립트

---

## SLIDE 1 — 표지

**Synapse-V**
백화점 통합 IT 모니터링 시스템

> *발표자 오프닝*
> "안녕하십니까. 오늘은 당사 IT 운영 방식을 근본적으로 바꿀 통합 모니터링 시스템, Synapse-V를 소개드리겠습니다."

---

## SLIDE 2 — WHY | 왜 만들었나

**현재 우리가 겪는 문제**

| 현실 | 결과 |
|---|---|
| 수십 개 시스템, 수십 명 담당자가 각자 모니터링 | 장애 인지까지 평균 15~30분 소요 |
| 로그 파일 수동 확인 → 원인 파악 지연 | 고객 불편·매출 손실로 직결 |
| 폐쇄망 환경 → 외부 SaaS 사용 불가 | 검증된 클라우드 솔루션 도입 불가 |
| 담당자가 퇴근 후·주말에도 수동 확인 | 운영 피로 누적, 야간 장애 대응 공백 |

> "지금까지 우리 팀은 장애가 나면 직접 서버에 붙어서 로그를 뒤지고, 단체 카톡으로 상황을 전파했습니다. 그 사이 고객 민원은 쌓이고 있었습니다."

---

## SLIDE 3 — WHAT | 무엇을 만들었나

**Synapse-V: 백화점 폐쇄망 전용 AI 통합 모니터링 시스템**

```
장애 발생 → AI 자동 감지 → 담당자 즉시 알림 → 해결책 제안
        단 5분 이내
```

- **설치 장벽 없음**: 대상 서버에 API 호출 한 번으로 에이전트 자동 설치
- **폐쇄망 100% 지원**: 인터넷 없이 내부망만으로 완전 동작
- **내부 LLM 연동**: 외부로 로그 데이터 유출 없이 AI 분석
- **AI 챗봇 내장**: 운영자가 자연어로 시스템 상태를 질의하고 즉시 답변

> "한 마디로 정의하면, 잠들어 있어도 IT 장애를 먼저 발견하고, 담당자에게 원인과 해결책을 들고 찾아오는 시스템입니다."

---

## SLIDE 4 — HOW | 어떻게 동작하나

**3단계 자동화 파이프라인**

```
[1단계] 수집
  대상 서버
  ├── CPU · 메모리 · 디스크 · 네트워크
  ├── 프로세스 상태
  ├── 애플리케이션 로그 (ERROR · CRITICAL · Exception)
  └── DB 성능 지표 (Oracle · PostgreSQL · MSSQL · MySQL)
       ↓ (Rust 에이전트 → Prometheus Remote Write)

[2단계] 분석
  Prometheus (메트릭 저장)
       ↓
  AI 분석 엔진 (log-analyzer)
  ├── 내부 LLM → 로그 이상 원인 분류
  ├── 벡터 DB  → 과거 유사 장애 검색
  └── 심각도 판정 (info / warning / critical)
       ↓

[3단계] 알림 & 대응
  Teams 채널 → 담당 팀 즉시 알림
  ├── 장애 원인 요약
  ├── 유사 장애 해결책 첨부
  └── "해결책 등록" 버튼 → 지식 베이스 축적

  + AI 챗봇
  └── 운영자가 자연어로 현황 질의 → 실시간 답변
```

> "수집부터 알림까지 사람 개입 없이 5분 주기로 자동 반복됩니다. 새벽 3시 장애도 담당자 핸드폰으로 바로 전달됩니다."

## SLIDE 5 — 주요 기능 ①

### 기능 1. AI 로그 분석 — "로그를 읽어주는 AI"

**기존 방식**: 운영자가 수백 줄 로그를 직접 확인 → 원인 파악 20~30분

**Synapse-V**: 내부 LLM이 5분마다 자동 분석

```
로그 패턴 감지
  ↓
원인 분류: "DB Connection Pool 고갈 — 세션 수 임계 초과"
심각도: CRITICAL
담당자: 홍길동 팀장 (Teams @mention)
```

**효과**: 원인 파악 시간 → 수십 분에서 수 초로 단축

---

## SLIDE 6 — 주요 기능 ②

### 기능 2. 유사 장애 검색 — "같은 장애, 두 번 고생하지 않는다"

**문제**: 같은 유형의 장애가 반복되어도 해결책이 개인 경험에 머묾

**Synapse-V**: 벡터 유사도 검색으로 과거 장애 이력 + 해결책 자동 제안

```
현재 장애 발생
  ↓
벡터 DB 검색 → 유사도 87% 매칭
  ↓
"2025-11-03 같은 증상 — 해결책: WAS 재기동 후 Connection Pool 설정 조정"
```

**효과**: 신규 투입 인력도 베테랑 수준의 대응 가능, 조직 지식 축적

## SLIDE 7 — 주요 기능 ③

### 기능 3. 에이전트 자동 배포 — "설치가 가장 쉬운 모니터링"

**기존 방식**: 서버마다 직접 SSH 접속 → 패키지 설치 → 설정 파일 작성 → 서비스 등록 → 수일 소요

**Synapse-V**: UI에서 버튼 클릭 한 번으로 자동 완료

```
관리 UI에서 대상 서버 정보 입력
  ↓
시스템이 자동으로:
  1. 서버에 SSH 접속
  2. 경량 수집 에이전트(4MB) 전송
  3. 모니터링 설정 파일 생성 (수집 항목·임계값 포함)
  4. 에이전트 실행 및 상태 확인
  ↓
완료 — 즉시 메트릭 수집 시작
```

**수집 항목**: CPU · 메모리 · 디스크 · 네트워크 · 프로세스 · 애플리케이션 로그

**DB 서버**: Oracle · PostgreSQL · MSSQL · MySQL 커넥션 정보만 입력하면  
세션 수 · TPS · 슬로우 쿼리 · 캐시 히트율 · 복제 지연 자동 수집

**효과**: 서버 50대 배포에 수일 → 반나절, 별도 설치 인력 불필요

## SLIDE 8 — 주요 기능 ④

### 기능 4. AI 운영 챗봇 — "대시보드 볼 시간도 없을 때"

**상황**: 장애 대응 중 여러 화면을 오가며 정보를 취합하는 시간이 없을 때

**Synapse-V 챗봇**: 자연어로 질문하면 즉시 답변

```
운영자: "지금 critical 상태인 시스템 뭐야?"
챗봇:   "현재 CRITICAL: 고객경험시스템(WAS1), 물류관리시스템(DB)
         - 고객경험: DB Connection Pool 고갈 (22분 전)
         - 물류관리: 슬로우 쿼리 급증 (8분 전)"

운영자: "고객경험 시스템 담당자 누구야?"
챗봇:   "홍길동 팀장 (gildong@company.com)
         마지막 로그인: 오늘 08:42"

운영자: "최근 일주일 장애 패턴 분석해줘"
챗봇:   "매주 화·목 오전 10~11시 집중 발생
         공통 원인: 배치 작업과 사용자 트래픽 피크 시간대 충돌"
```

- **폐쇄망 완전 지원**: 내부 LLM만 사용, 외부 인터넷 연결 불필요
- **권한 통제**: 로그인 사용자 권한에 따라 조회 범위 제한
- **대화 이력**: 세션 내 대화 맥락 유지 → 추가 질문에도 문맥 파악

**효과**: 대시보드 없이도 현황 파악 가능, 신규 운영자 온보딩 기간 단축

## SLIDE 9 — 주요 기능 ⑤

### 기능 5. Teams 실시간 알림 + 해결책 등록 — "알림이 끝이 아닌 시작"

기존 모니터링 도구의 알림은 "문제가 났다"는 통보에서 끝.

**Synapse-V Teams 카드**:

```
🔴 [CRITICAL] 고객경험시스템 / WAS1
원인: DB Connection Pool 고갈
유사 장애: 87% 일치 (2025-11-03)
해결책: WAS 재기동 → Pool 설정 조정

[해결책 등록] 버튼  ← 해결 후 클릭 → 지식 베이스 자동 저장
```

- 시스템별 담당 Teams 채널로 분리 발송
- 복구 알림도 자동 발송 (담당자가 일일이 "해결됐습니다" 메시지 불필요)

**효과**: 장애 대응 커뮤니케이션 비용 60% 절감 (추정)

## SLIDE 10 — 주요 기능 ⑥

### 기능 6. 계층적 집계 & 장애 예방 분석 — "다음 장애를 미리 본다"

단순 실시간 알림을 넘어, **패턴 기반 예방**

| 주기 | 분석 내용 |
|---|---|
| 1시간 | 메트릭 이상 패턴 + AI 단기 예측 |
| 1일 | 일별 장애 빈도 + 피크 시간대 분석 |
| 1주 | 주간 트렌드 + 반복 이상 항목 식별 |
| 1월 | 월간 리포트 + 중장기 용량 예측 |

**예방 알림 예시**:

```
⚠️ 트렌드 감지: 고객경험시스템 CPU 사용률
최근 3일 연속 업무 시간대 92% 초과
→ 주말 전 사전 증설 권고
```

**효과**: 사후 대응에서 사전 예방으로 전환

## SLIDE 12 — 기대 효과 & 다음 단계

**정량적 기대 효과**

| 지표 | 현재 | 목표 |
|---|---|---|
| 장애 인지 시간 | 15~30분 | 5분 이내 |
| 원인 파악 시간 | 20~60분 | 수 초 (AI 분석) |
| 야간 장애 대응 | 담당자 수동 확인 | 자동 알림 |

**현재 구현 완료 범위**

- 메트릭 수집 · 알림 · Grafana 시각화

- AI 운영 챗봇 (자연어 질의 · 도구 연동)
- 웹 관리 UI (실시간 대시보드 포함)
- 분산 추적 (OTel + Tempo)

1. 파일럿 대상 시스템 선정 (2~3개 권장)
