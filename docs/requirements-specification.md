# 백화점 통합 모니터링 시스템 (Synapse-V) - 요구사항 명세서

> **문서 버전**: v1.0
> **작성일**: 2026-03-08
> **상태**: 요구사항 확정 (설계 전 단계)

---

## 1. 프로젝트 개요

### 1.1 목적
백화점에서 운영 중인 13개 이기종 시스템에 대한 **통합 모니터링 시스템**을 구축하여, 시스템 자원 상태 감시, 로그 수집/분석, 이상 징후 탐지 및 담당자 알림을 자동화한다.

### 1.2 현행 환경 요약

| 항목 | 현황 |
|------|------|
| 대상 시스템 | 약 13개 (웹/앱 서비스, DB, 미들웨어) |
| 인프라 | 물리 서버 / VM |
| OS | Linux + Windows 혼합 |
| 기술 스택 | Java/Spring + Tomcat, Nginx/Apache |
| DB | Oracle, PostgreSQL, MSSQL |
| 네트워크 | 동일 네트워크, 특별한 보안 제약 없음 |
| 기존 모니터링 | 없음 (신규 구축) |
| 운영 체계 | 시스템별 담당자 지정 |

## 2. 기능적 요구사항 (Functional Requirements)

### FR-01: 시스템 자원 모니터링
- 13개 시스템의 CPU, Memory, Disk 사용률을 실시간 수집한다.
- 수집 주기: 최소 1분 (자원 메트릭)
- Linux와 Windows 양쪽 OS에서 동작하는 에이전트를 통해 수집한다.

### FR-02: 임계값 기반 자동 알림
- CPU, Memory, Disk 사용률이 **80% 이상**일 때 해당 시스템 담당자에게 자동 알림을 발송한다.
- 알림 채널: Slack/Teams 메시지 + Webhook (커스텀 연동)
- 알림에는 다음 정보가 포함되어야 한다:
  - 시스템명, 호스트명
  - 초과된 메트릭 종류 및 현재 값
  - 발생 시각
- 반복 알림 방지 (동일 조건 알림 쿨다운 설정 가능)

### FR-03: 로그 수집
- 13개 시스템의 **파일 기반 로그**를 수집한다.
- 수집 주기: **5분 단위**
- 각 시스템의 로그 파일 경로를 설정으로 관리할 수 있어야 한다.
- 증분 수집 (마지막 수집 이후 추가된 로그만 수집)

### FR-04: LLM 기반 로그 이상 분석
- 수집된 로그를 **내부 LLM API**를 호출하여 분석한다. (별도 내부 LLM 서버 활용)
- 분석 대상: ERROR, WARN, Exception, 비정상 패턴
- 분석 결과 이상이 감지되면 해당 시스템 담당자에게 알림을 발송한다.
- 알림에는 다음 정보가 포함되어야 한다:
  - 원본 로그 내용 (요약)
  - LLM 분석 결과 (원인 추정, 심각도)
  - 시스템명, 발생 시각

### FR-05: 통합 모니터링 대시보드
- 13개 시스템의 전체 상태를 한눈에 볼 수 있는 대시보드를 제공한다.
- 대시보드에 표시할 항목:
  - 시스템별 자원 사용률 (CPU/Memory/Disk) 게이지
  - 시스템별 상태 (정상/경고/위험) 표시
  - 최근 알림 이력
  - 로그 분석 결과 요약
  - 시스템별 상세 뷰 (드릴다운)

### FR-06: 시스템별 담당자 관리
- 각 시스템에 1명 이상의 담당자를 매핑할 수 있어야 한다.
- 담당자별 알림 수신 채널을 설정할 수 있어야 한다.
- 담당자 정보 변경 시 즉시 반영되어야 한다.

## 3. 추가 수집 항목 (사용자 요청에 의한 확장 메트릭)

기본 자원(CPU/Memory/Disk) 외에 다음 항목의 수집을 권장한다:

### 3.1 시스템/인프라 메트릭
| 항목 | 설명 | 대상 |
|------|------|------|
| **네트워크 I/O** | 대역폭, 패킷 손실률, 지연시간 | 전체 |
| **디스크 I/O** | 읽기/쓰기 속도, IOPS, 대기 시간 | 전체 |
| **시스템 Uptime** | 시스템 가동 시간, 마지막 재시작 시간 | 전체 |
| **프로세스 상태** | 주요 프로세스/서비스 alive/dead 여부 | 전체 |
| **TCP 연결 수** | ESTABLISHED, TIME_WAIT 등 연결 상태 | 전체 |

### 3.2 WAS/애플리케이션 메트릭
| 항목 | 설명 | 대상 |
|------|------|------|
| **Tomcat 스레드 풀** | 활성 스레드 수, 최대 스레드, 대기 큐 | Java/Spring 시스템 |
| **HTTP 응답 시간** | 평균/p95/p99 응답 시간 | 웹 서비스 |
| **HTTP 상태 코드** | 2xx/4xx/5xx 비율 | 웹 서비스 |
| **요청 처리량 (TPS)** | 초당 트랜잭션 수 | 웹 서비스 |
| **세션 수** | 활성 세션 수 | 웹 서비스 |
| **JVM 메트릭** | 힙 메모리, GC 횟수/시간, 스레드 수 | Java 시스템 |

### 3.3 DB 메트릭
| 항목 | 설명 | 대상 |
|------|------|------|
| **커넥션 풀** | 활성/대기/최대 커넥션 수 | Oracle, PostgreSQL, MSSQL |
| **슬로우 쿼리** | 설정 시간 초과 쿼리 수 및 내용 | 전체 DB |
| **테이블스페이스** | 사용률, 남은 공간 | Oracle |
| **Lock/Deadlock** | 락 대기, 데드락 발생 횟수 | 전체 DB |
| **Replication Lag** | 복제 지연 시간 (해당 시) | 해당 DB |

### 3.4 웹서버 메트릭
| 항목 | 설명 | 대상 |
|------|------|------|
| **Nginx 활성 연결** | 현재 활성/대기/처리 중 연결 수 | Nginx |
| **Apache 워커 상태** | Busy/Idle 워커 수 | Apache |
| **요청률** | 초당 요청 수 | Nginx/Apache |

## 4. 비기능적 요구사항 (Non-Functional Requirements)

### NFR-01: 배포 및 운영
- 모니터링 시스템은 **Docker 기반**으로 구축한다.
- 별도 확보된 전용 서버에 배포한다.
- docker-compose 또는 유사한 오케스트레이션으로 관리한다.

### NFR-02: 확장성
- 향후 시스템 추가 시 설정만으로 모니터링 대상을 확장할 수 있어야 한다.
- 수집 항목도 플러그인/설정 방식으로 확장 가능해야 한다.

### NFR-03: 가용성
- 모니터링 시스템 자체의 장애를 감지할 수 있어야 한다 (Self-monitoring).
- 단일 장애점(SPOF)을 최소화한다.

### NFR-04: 데이터 보존
- 메트릭 데이터: 최소 30일 보존 (설정 가능)
- 로그 데이터: 최소 7일 보존 (설정 가능)
- 알림 이력: 최소 90일 보존

### NFR-05: 보안
- LLM은 내부 API를 사용하여 로그 데이터가 외부로 유출되지 않도록 한다.
- 대시보드 접근은 인증/인가를 통해 보호한다.
- 로그 내 개인정보가 있을 경우 마스킹 처리 방안을 고려한다.

### NFR-06: 성능
- 메트릭 수집~대시보드 반영: 2분 이내
- 알림 발송: 임계값 초과 감지 후 1분 이내
- 대시보드 로딩: 5초 이내

## 5. 사용자 스토리

### US-01: 자원 모니터링 및 알림
> **담당자로서**, 내가 관리하는 시스템의 CPU/Memory/Disk가 80%를 넘으면 **즉시 Slack/Teams 알림을 받고 싶다**. 그래야 장애가 발생하기 전에 선제 대응할 수 있다.

**인수 조건**:
- [ ] CPU/Memory/Disk 각각의 사용률이 80% 이상이 되면 알림 발송
- [ ] 알림에 시스템명, 메트릭 종류, 현재 값, 시각이 포함됨
- [ ] 동일 조건 반복 알림은 쿨다운(최소 5분) 적용
- [ ] Slack/Teams 채널과 Webhook 모두로 알림 수신 가능

### US-02: 로그 수집 및 LLM 분석
> **담당자로서**, 내가 관리하는 시스템의 로그에서 에러나 이상 패턴이 발견되면 **LLM이 분석한 결과와 함께 알림을 받고 싶다**. 그래야 로그를 직접 분석하는 시간을 줄일 수 있다.

**인수 조건**:
- [ ] 5분 주기로 각 시스템의 로그 파일에서 신규 로그를 수집
- [ ] 수집된 로그 중 ERROR/WARN/Exception 패턴을 LLM으로 분석
- [ ] 분석 결과에 원인 추정, 심각도가 포함됨
- [ ] 이상 감지 시 해당 시스템 담당자에게 알림 발송

### US-03: 통합 대시보드
> **운영 관리자로서**, 13개 시스템의 전체 상태를 **하나의 대시보드에서 한눈에 파악하고 싶다**. 필요 시 특정 시스템의 상세 정보로 드릴다운할 수 있어야 한다.

**인수 조건**:
- [ ] 전체 시스템 목록과 각각의 상태(정상/경고/위험)가 표시됨
- [ ] 시스템별 CPU/Memory/Disk 게이지 차트 표시
- [ ] 최근 알림 이력 표시
- [ ] 시스템 클릭 시 상세 메트릭/로그 뷰로 이동
- [ ] 대시보드 자동 새로고침 (1분 주기)

### US-04: 담당자 관리
> **운영 관리자로서**, 각 시스템의 담당자를 **설정하고 변경할 수 있어야 한다**. 담당자별로 알림 채널을 다르게 설정할 수 있어야 한다.

**인수 조건**:
- [ ] 시스템-담당자 매핑을 UI에서 설정 가능
- [ ] 담당자별 알림 채널(Slack/Teams/Webhook) 개별 설정
- [ ] 변경 즉시 반영

### US-05: 추가 메트릭 모니터링
> **담당자로서**, 기본 자원 외에 **WAS 상태, DB 커넥션, HTTP 응답시간 등 애플리케이션 레벨 메트릭**도 모니터링하고 싶다.

**인수 조건**:
- [ ] Tomcat 스레드/세션, JVM 메트릭 수집
- [ ] DB 커넥션 풀, 슬로우 쿼리 수집
- [ ] HTTP 응답시간, 상태코드 분포 수집
- [ ] 각 메트릭에 대한 임계값 설정 및 알림 가능

## 6. 기술적 제약 및 고려사항

### 6.1 에이전트 호환성
- Linux와 Windows 양쪽에서 동작하는 에이전트가 필요하다.
- 에이전트는 대상 시스템에 최소한의 부하만 주어야 한다.

### 6.2 LLM 연동
- 별도 내부 LLM API를 HTTP로 호출하여 분석한다. (Ollama 미사용)
- 모니터링 서버에는 LLM 관련 리소스가 필요하지 않다.
- 내부 LLM API의 응답 시간과 처리량(throughput)에 따라 분석 배치 크기를 조절해야 한다.
- 13개 시스템의 5분 주기 로그를 처리할 수 있는 API 호출 한도를 확인해야 한다.

### 6.3 DB 다양성
- Oracle, PostgreSQL, MSSQL 각각에 대한 메트릭 수집기(exporter)가 필요하다.
- DB별로 수집 가능한 메트릭과 접근 방식이 다르므로 개별 설정이 필요하다.

### 6.4 로그 형식 표준화
- 파일 기반 로그의 형식이 시스템마다 다를 수 있다.
- 로그 수집 시 파싱 규칙을 시스템별로 설정할 수 있어야 한다.
