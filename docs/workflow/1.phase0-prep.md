## 폐쇄망 사전 준비 (macOS에서 수행)

> 모든 설치 파일과 Docker 이미지를 인터넷 연결된 Mac에서 먼저 준비한 뒤, 모니터링 서버 및 대상 서버로 전송합니다.

### 1. Docker 이미지 Pull & Save (Mac)

```bash
# ⚠️ 반드시 --platform linux/amd64 명시
# Mac(Apple Silicon)은 arm64이므로 누락 시 서버에서 실행 불가
docker pull --platform linux/amd64 prom/prometheus:v3.10.0
docker pull --platform linux/amd64 prom/alertmanager:v0.31.1
docker pull --platform linux/amd64 grafana/loki:3.6.7
docker pull --platform linux/amd64 grafana/grafana:12.4.0
docker pull --platform linux/amd64 postgres:16-alpine

# 아키텍처 확인 (저장 전 반드시 검증)
docker inspect prom/prometheus:v3.10.0 | grep Architecture
# → "Architecture": "amd64"  ← 반드시 amd64 확인

# 이미지를 tar 파일로 저장
mkdir -p ~/aoms-offline/docker-images

docker save prom/prometheus:v3.10.0       | gzip > ~/aoms-offline/docker-images/prometheus.tar.gz
docker save prom/alertmanager:v0.31.1     | gzip > ~/aoms-offline/docker-images/alertmanager.tar.gz
docker save grafana/loki:3.6.7            | gzip > ~/aoms-offline/docker-images/loki.tar.gz
docker save grafana/grafana:12.4.0        | gzip > ~/aoms-offline/docker-images/grafana.tar.gz
docker save postgres:16-alpine            | gzip > ~/aoms-offline/docker-images/postgres.tar.gz

# ⚠️ 버전별 주요 변경사항 / 상호 호환성 메모
#
# [Prometheus v3.10.0]
#   - v2.x → v3.x 주요 변경: 기존 설정 파일은 대부분 호환
#   - deprecated된 storage.tsdb.* 일부 플래그 제거됨
#   - web.yml Basic Auth 구조는 동일하게 사용 가능
#   - Alertmanager v0.27+ API v1 완전 제거 → v0.31.1 함께 사용 필수
#
# [Alertmanager v0.31.1]
#   - Prometheus v3와 완전 호환
#   - UTF-8 strict 모드 기본 지원
#   - Teams webhook 방식 동일하게 사용 가능
#
# [Grafana 12.4.0]
#   - 이미지명 변경: grafana/grafana-oss → grafana/grafana (v12.4.0부터)
#   - Prometheus v3, Alertmanager v0.31.1, Loki v3.6.7 모두 데이터소스 지원
#   - UID 472 동일 적용
#
# [Loki v3.6.7]
#   - Grafana 12.x와 완전 호환
#   - 로그 수집 에이전트: Grafana Alloy v1.8.3 사용 (promtail 대체)
#   - BoltDB 스토리지 제거됨 → TSDB 기본 사용

# 커스텀 서비스 이미지 빌드 후 저장 (소스 작성 완료 후)
# docker build -t aoms-admin-api:1.0 ./services/admin-api
# docker build -t aoms-log-analyzer:1.0 ./services/log-analyzer
```

## 이중화 시스템 라벨링 규칙

> 이중화 구성(WAS 1번기/2번기 등)에서 **논리 시스템명**과 **서버 인스턴스**를 분리하여 관리합니다.

### 라벨 설계 원칙


| 라벨              | 설명                           | 예시                             |
| --------------- | ---------------------------- | ------------------------------ |
| `system_name`   | **논리 시스템명** — 대시보드/알림 라우팅 기준 | `customer-experience`          |
| `instance_role` | **서버 역할 및 번호** — 서버 구분       | `was1`, `was2`, `web1`, `web2` |
| `host`          | **실제 호스트명 또는 IP**            | `192.168.1.10`                 |


### 적용 예시 (고객경험 시스템)

```yaml
# Prometheus scrape_configs 예시
- job_name: 'customer-experience'
  static_configs:
    - targets: ['192.168.1.10:9100']
      labels:
        system_name: 'customer-experience'   # 논리 시스템명 (통합 대시보드/알림 기준)
        instance_role: 'was1'                # 1번기
        host: 'cx-was01'

    - targets: ['192.168.1.11:9100']
      labels:
        system_name: 'customer-experience'   # 동일 system_name
        instance_role: 'was2'                # 2번기
        host: 'cx-was02'

    - targets: ['192.168.1.12:9100']
      labels:
        system_name: 'customer-experience'
        instance_role: 'web1'
        host: 'cx-web01'

    - targets: ['192.168.1.13:9100']
      labels:
        system_name: 'customer-experience'
        instance_role: 'web2'
        host: 'cx-web02'
```

### 이중화 구성 PromQL 예시

```promql
# 특정 시스템 전체 서버의 CPU 평균
avg by(instance_role) (
  (1 - rate(node_cpu_seconds_total{mode="idle", system_name="customer-experience"}[5m])) * 100
)

# 알림 Rule: system_name 단위로 any 서버가 임계값 초과 시 발생
# → 알림 라벨에 instance_role이 포함되어 어느 서버인지 특정 가능
(1 - avg by(instance, system_name, instance_role) (
  rate(node_cpu_seconds_total{mode="idle"}[5m])
)) * 100 > 80
```

### Alloy 라벨 설정 (이중화 예시)

```alloy
// cx-was01 서버의 config.alloy
// Grafana Alloy v1.x — promtail.yml 대체

// ── Loki 전송 설정 ────────────────────────────────────────────
loki.write "default" {
  endpoint {
    url = "http://<monitoring-server>:3100/loki/api/v1/push"
  }
}

// ── 애플리케이션 로그 수집 ─────────────────────────────────────
local.file_match "app_logs" {
  path_targets = [{
    __path__      = "/app/logs/*.log",
    system_name   = "customer-experience",  // 논리 시스템명
    instance_role = "was1",                 // 서버 구분
    host          = "cx-was01",
    log_type      = "application",
    job           = "app-logs",
  }]
}

loki.source.file "app_logs" {
  targets    = local.file_match.app_logs.targets
  forward_to = [loki.process.app_logs.receiver]
}

loki.process "app_logs" {
  stage.multiline {
    firstline     = "^(\\d{4}[-/\\.]\\d{2}[-/\\.]\\d{2}|\\[\\d{4}|\\d{2}:\\d{2}:\\d{2})"
    max_wait_time = "3s"
    max_lines     = 200
  }
  stage.drop {
    expression = "^\\s*$"    // 빈줄 제거
  }
  stage.drop {
    expression = "(?i)^(?!.*(error|warn|fatal|exception|fail))"  // 에러만 전송
  }
  stage.regex {
    expression = "(?i)(?P<level>ERROR|WARN(?:ING)?|FATAL|CRITICAL|INFO|DEBUG)"
  }
  stage.labels {
    values = { level = "" }
  }
  forward_to = [loki.write.default.receiver]
}
```
