# Synapse-V — Server B: 벡터 DB 서버 구축 (Qdrant)

> ⚠️ **ADR-011/012 적용 (2026-04-23)**: 이 문서의 Ollama 관련 섹션(SB.2~SB.3 Ollama 이미지 로드/모델 복원, bge-m3 GGUF 등)은 **폐기**되었습니다.
> 임베딩은 log-analyzer 컨테이너 내 FastEmbed(ONNX) 인프로세스로 이관됨. Server B는 **Qdrant 전용**입니다.
> 운영 배포는 `deploy-guide.md` 섹션 2를 참조하세요. 이 문서의 Qdrant 관련 절차만 유효합니다.

> **관련 단계**: Phase 4b 벡터 DB 도입 전 필수 선행 작업
> **서버 사양**: 2코어 / 16GB RAM / 50GB Disk (RedHat 8.9)
> **이전 단계**: [Phase 4 LLM 분석](./phase4-llm.md)
> **다음 단계**: [Phase 4b 벡터 DB](./phase4b-vector.md)

### SB.1 — Server B 환경 준비 (RedHat 8.9)

```bash
# ---- Server B (RedHat 8.9)에서 실행 ----

# Docker 설치 (폐쇄망: RPM 패키지 사전 전송 필요)
# Mac에서 RPM 다운로드 후 전송
#   curl -LO https://download.docker.com/linux/rhel/8/x86_64/stable/Packages/docker-ce-26.1.4-1.el8.x86_64.rpm
#   (containerd, docker-ce-cli 등 의존 패키지도 함께 다운로드)
# 전송 후 설치:
sudo rpm -ivh /opt/aoms-offline/rpms/containerd.io-*.rpm
sudo rpm -ivh /opt/aoms-offline/rpms/docker-ce-cli-*.rpm
sudo rpm -ivh /opt/aoms-offline/rpms/docker-ce-*.rpm

sudo systemctl enable --now docker

# Docker Compose Plugin 설치
sudo mkdir -p /usr/local/lib/docker/cli-plugins
sudo cp /opt/aoms-offline/rpms/docker-compose-linux-x86_64 \
        /usr/local/lib/docker/cli-plugins/docker-compose
sudo chmod +x /usr/local/lib/docker/cli-plugins/docker-compose

# 현재 사용자를 docker 그룹에 추가
sudo usermod -aG docker $USER
newgrp docker

# SELinux 상태 확인 (Enforcing이어야 함)
getenforce
# → Enforcing  ← Docker 볼륨 마운트 시 :z 옵션 필수

# 시간대 설정 확인
sudo timedatectl set-timezone Asia/Seoul
timedatectl status

# 디스크 여유 공간 확인
df -h   # / 파티션 50GB 이상 확인

# Server B 전용 디렉토리 구조 생성
mkdir -p /opt/serverb/ollama-models   # Ollama 모델 영구 저장소
mkdir -p /opt/serverb/qdrant-storage  # Qdrant 데이터 (docker volume 미사용 시 참고)
mkdir -p /opt/serverb/compose         # docker-compose.yml 위치
mkdir -p /var/log/serverb             # 컨테이너 로그 수집용

# 방화벽 설정: Server A IP에서만 Ollama(11434), Qdrant(6333/6334) 접근 허용
# SERVER_A_IP 를 실제 Server A의 IP로 교체하세요 (예: 10.0.1.10)
SERVER_A_IP="10.0.1.10"

sudo firewall-cmd --permanent --add-rich-rule="rule family=\"ipv4\" source address=\"${SERVER_A_IP}/32\" port port=\"11434\" protocol=\"tcp\" accept"   # Ollama
sudo firewall-cmd --permanent --add-rich-rule="rule family=\"ipv4\" source address=\"${SERVER_A_IP}/32\" port port=\"6333\" protocol=\"tcp\" accept"    # Qdrant HTTP
sudo firewall-cmd --permanent --add-rich-rule="rule family=\"ipv4\" source address=\"${SERVER_A_IP}/32\" port port=\"6334\" protocol=\"tcp\" accept"    # Qdrant gRPC
sudo firewall-cmd --reload
firewall-cmd --list-rich-rules   # 적용 확인
```

> **주의**: Ollama(11434)와 Qdrant(6333/6334)는 Server A IP에서만 접근 가능하도록
> 제한합니다. `0.0.0.0/0` 개방은 폐쇄망 정책 위반입니다.

### SB.3 — Ollama 배포 및 bge-m3 모델 로드

#### docker-compose.yml (Server B 전용)

`**/opt/serverb/compose/docker-compose.yml**`:

```yaml
services:
  ollama:
    image: ollama/ollama:latest
    container_name: aoms-ollama
    ports:
      - "11434:11434"
    volumes:
      - /opt/serverb/ollama-models:/root/.ollama
    deploy:
      resources:
        limits:
          memory: 2048M
          cpus: '0.5'
    restart: unless-stopped
    networks:
      - serverb-network

  qdrant:
    image: qdrant/qdrant:v1.9.0
    container_name: aoms-qdrant
    ports:
      - "6333:6333"
      - "6334:6334"
    volumes:
      - qdrant_data:/qdrant/storage:z
    environment:
      - QDRANT__SERVICE__GRPC_PORT=6334
    deploy:
      resources:
        limits:
          memory: 8192M
          cpus: '1.0'
    restart: unless-stopped
    networks:
      - serverb-network

networks:
  serverb-network:
    driver: bridge

volumes:
  qdrant_data:
```

> **메모리 배분 근거** (16GB 총량):
> - Ollama + bge-m3 모델: 2GB (임베딩 전용, GPU 없음)
> - Qdrant: 8GB (벡터 인덱스 + on-disk 오버플로)
> - OS/기타: 6GB 여유

#### Ollama 모델 로드 (폐쇄망: 사전 준비된 모델 파일 복원)

```bash
# 사전 준비된 bge-m3 모델 파일 복원
# (인터넷 PC에서: docker exec ollama ollama pull bge-m3 후 ~/.ollama 폴더 압축)
mkdir -p /opt/serverb/ollama-models
tar xzf /opt/aoms-offline/ollama-models.tar.gz -C /opt/serverb/
# → /opt/serverb/ollama-models/models/ 하위에 blobs/, manifests/ 배치

# 컨테이너 시작 (Ollama 단독 먼저 기동)
cd /opt/serverb/compose
docker compose up -d ollama

# 컨테이너 정상 기동 확인
docker ps | grep ollama
# CONTAINER ID   IMAGE                  STATUS         PORTS
# xxxxxxxxxxxx   ollama/ollama:latest   Up X seconds   0.0.0.0:11434->11434/tcp

# 모델 목록 확인 (복원된 bge-m3가 표시되어야 함)
curl http://localhost:11434/api/tags
# → {"models":[{"name":"bge-m3","size":...,"details":{"family":"bert",...}}]}

# 임베딩 동작 테스트 (1024차원 벡터 반환 확인)
curl -X POST http://localhost:11434/api/embeddings \
  -H "Content-Type: application/json" \
  -d '{"model": "bge-m3", "prompt": "ORA-00060: deadlock detected waiting for resource"}'
# → {"embedding":[...1024개 float값...]}
```

> **bge-m3 모델 사전 준비 (인터넷 연결 PC에서)**:
>
> ```bash
> # 인터넷 PC에서 Ollama 실행 후 모델 pull
> docker run -d --name temp-ollama ollama/ollama:latest
> docker exec temp-ollama ollama pull bge-m3
>
> # ~/.ollama 폴더 압축 (컨테이너 내부)
> docker exec temp-ollama tar czf /tmp/ollama-models.tar.gz -C /root .ollama
> docker cp temp-ollama:/tmp/ollama-models.tar.gz ./ollama-models.tar.gz
>
> # Server B로 전송
> scp ollama-models.tar.gz user@SERVER_B_IP:/opt/aoms-offline/
> ```

### SB.4 — Qdrant 배포 및 컬렉션 생성

```bash
# Qdrant 컨테이너 시작
cd /opt/serverb/compose
docker compose up -d qdrant

# 컨테이너 정상 기동 확인
docker ps | grep qdrant
# CONTAINER ID   IMAGE                  STATUS         PORTS
# xxxxxxxxxxxx   qdrant/qdrant:v1.9.0   Up X seconds   0.0.0.0:6333->6333/tcp, 0.0.0.0:6334->6334/tcp

# 헬스체크
curl http://localhost:6333/healthz
# → OK

# log_incidents 컬렉션 생성 (int8 양자화 + on-disk)
# - vectors.size: bge-m3 임베딩 차원 (1024)
# - on_disk: true → RAM 초과 시 디스크 사용 (16GB 서버 필수 설정)
# - int8 양자화 → 벡터 저장 용량 ~4배 절감, 검색 속도 향상
curl -X PUT http://localhost:6333/collections/log_incidents \
  -H "Content-Type: application/json" \
  -d '{
    "vectors": {
      "size": 1024,
      "distance": "Cosine",
      "on_disk": true
    },
    "hnsw_config": {
      "on_disk": true,
      "m": 16,
      "ef_construct": 100
    },
    "quantization_config": {
      "scalar": {
        "type": "int8",
        "quantile": 0.99,
        "always_ram": false
      }
    }
  }'
# → {"result":true,"status":"ok"}

# 페이로드 인덱스 생성 (빠른 필터링을 위해)
# system_name: 시스템별 격리 검색
curl -X PUT http://localhost:6333/collections/log_incidents/index \
  -H "Content-Type: application/json" \
  -d '{"field_name": "system_name", "field_schema": "keyword"}'

# severity: CRITICAL/ERROR/WARN 필터링
curl -X PUT http://localhost:6333/collections/log_incidents/index \
  -H "Content-Type: application/json" \
  -d '{"field_name": "severity", "field_schema": "keyword"}'

# instance_role: 이중화 서버 역할별 필터링 (was1/was2/db-primary 등)
curl -X PUT http://localhost:6333/collections/log_incidents/index \
  -H "Content-Type: application/json" \
  -d '{"field_name": "instance_role", "field_schema": "keyword"}'

# 컬렉션 정보 확인
curl http://localhost:6333/collections/log_incidents
# → {"result":{"status":"green","vectors_count":0,"config":{"params":{"vectors":{"size":1024,...}}}}}
```

> **int8 양자화 설정 근거**:
> - `quantile: 0.99` → 상위 1% 이상치를 클리핑하여 정밀도 손실 최소화
> - `always_ram: false` → RAM 부족 시 디스크 폴백 (16GB 서버에서 필수)
> - 코사인 유사도(Cosine)는 벡터 길이 정규화 불필요 → bge-m3 권장 거리 함수

```bash
# ⚠️ Server A IP가 다른 경우 기존 룰 제거 후 재등록
# sudo firewall-cmd --permanent --remove-rich-rule='rule family="ipv4" ...'
# sudo firewall-cmd --reload
```

#### Server A에서 Server B 접근 테스트

```bash
# ---- Server A (모니터링 서버)에서 실행 ----
SERVER_B_IP="10.0.1.20"   # Server B 실제 IP로 교체

# Ollama 포트 연결 확인
curl --connect-timeout 5 http://${SERVER_B_IP}:11434/api/tags
# → {"models":[...]} 응답이면 정상

# Qdrant 포트 연결 확인
curl --connect-timeout 5 http://${SERVER_B_IP}:6333/healthz
# → OK 응답이면 정상

# 연결 실패 시 원인 확인
# curl: (7) Failed to connect → Server B 방화벽 룰 미적용 또는 IP 오기입
# curl: (28) Operation timed out → 네트워크 경로 문제 (라우팅/스위치 설정 확인)
```

### SB.6 — 통합 연결 테스트 (Server A에서)

Server A → Server B 원격 호출로 임베딩 생성, 벡터 저장, 유사도 검색까지 E2E 확인합니다.

```bash
# ---- Server A에서 실행 ----
SERVER_B_IP="10.0.1.20"   # Server B 실제 IP로 교체

# 1. Ollama 임베딩 원격 호출 테스트
curl -X POST http://${SERVER_B_IP}:11434/api/embeddings \
  -H "Content-Type: application/json" \
  -d '{"model": "bge-m3", "prompt": "테스트 로그 메시지"}'
# → {"embedding":[0.0123, -0.0456, ...]}  (1024개 float)

# 2. 테스트 포인트 저장 (임베딩 결과를 EMBEDDING_VECTOR 변수에 담아 사용)
#    아래 <EMBEDDING_VECTOR> 자리에 위 응답의 embedding 배열을 붙여 넣으세요
curl -X PUT http://${SERVER_B_IP}:6333/collections/log_incidents/points \
  -H "Content-Type: application/json" \
  -d '{
    "points": [{
      "id": "00000000-0000-0000-0000-000000000001",
      "vector": [<EMBEDDING_VECTOR>],
      "payload": {
        "system_name": "test",
        "severity": "warning",
        "instance_role": "was1",
        "log_pattern": "테스트 로그 패턴",
        "timestamp": "2026-03-15T00:00:00Z"
      }
    }]
  }'
# → {"result":{"operation_id":0,"status":"completed"},"status":"ok"}

# 3. 유사 벡터 검색 (저장한 벡터와 동일 벡터로 검색 → score 1.0 반환 확인)
curl -X POST http://${SERVER_B_IP}:6333/collections/log_incidents/points/search \
  -H "Content-Type: application/json" \
  -d '{
    "vector": [<EMBEDDING_VECTOR>],
    "limit": 3,
    "with_payload": true,
    "score_threshold": 0.75
  }'
# → {"result":[{"id":"00000000-...","score":1.0,"payload":{"system_name":"test",...}}]}
# 저장한 테스트 포인트가 score 1.0으로 반환되면 성공

# 4. 테스트 데이터 정리
curl -X POST http://${SERVER_B_IP}:6333/collections/log_incidents/points/delete \
  -H "Content-Type: application/json" \
  -d '{"points": ["00000000-0000-0000-0000-000000000001"]}'
# → {"result":{"operation_id":1,"status":"completed"},"status":"ok"}
```
